# Why We Exist

## The problem

People often want to do something in real life, but the path from intention to attendance breaks.

The failure usually does not happen because people hate real life.

It happens because the coordination layer is weak:

- plans are scattered across chats;
- event details are unclear;
- joining feels awkward;
- nobody knows who else is going;
- beginners fear arriving alone;
- organizers repeat the same information manually;
- people scroll instead of committing;
- the chat does not reliably become a real meeting.

GO IRL exists to fix this broken path.

## The product answer

GO IRL turns a vague social intention into a clear real-life plan.

The product should make this loop simple:

```text
create event -> share -> participants join -> event chat -> people show up in real life
```

Everything else is secondary.

## The mission

Less scrolling. More living.

This is not only a slogan. It is a product constraint.

If a feature increases screen time without improving real attendance, it should be rejected, delayed, or redesigned.

## The user promise

A user should be able to answer quickly:

- What is happening?
- When?
- Where?
- Who is going?
- Is there space for me?
- Can I join without awkward negotiation?
- Can I trust this enough to attend?

A user should not need to read a long chat thread to understand whether an event is real, open, and worth attending.

## The organizer promise

An organizer should be able to create a small real-life event quickly and share it through Telegram.

The organizer should not need to manually explain the same details again and again.

GO IRL should help the organizer communicate:

- event type;
- time;
- place;
- capacity;
- participant status;
- join rules;
- event chat;
- trust signals.

## The social promise

GO IRL is not trying to maximize posts, likes, comments, or screen time.

The product should maximize successful real-life meetings.

Success means:

```text
Someone saw an event, joined it, coordinated briefly, and actually came.
```

## Why Telegram first

People already coordinate informal events in Telegram.

GO IRL should not fight that behavior. It should make it structured enough to work.

Telegram-first means:

- event links are easy to share;
- Mini App opens fast;
- users do not need a heavy new account system for beta;
- event chat stays close to coordination;
- real-life planning stays inside an existing social channel.

## Why Olomouc first

Local density matters more than catalog size.

A small city with real repeat usage is better than a large empty platform.

Olomouc beta exists to validate:

- whether people create events;
- whether shared events bring joins;
- whether joins create chat activity;
- whether chat activity helps attendance;
- whether users return after attending.

## Why small events first

GO IRL is not initially built for massive public events.

Small events are better for MVP because they expose the real trust problem:

- Will I know where to go?
- Will someone meet me?
- Will I feel awkward?
- Will the event actually happen?
- Will people show up?

This is why clear cards, participant state, event chat, and Sport Coach matter more than broad event discovery before beta.

## What we deliberately avoid

Before closed beta, GO IRL must not become:

- a generic event calendar;
- a ticketing platform;
- a dating app;
- a social feed;
- a club CRM;
- a city media portal;
- a broad lifestyle super-app.

These directions may contain useful ideas, but they are not the current product.

## Foundation rule

GO IRL should always be judged by real-life conversion, not by content volume.

The core question is:

> Did this make it easier for people to meet in real life?

If not, it does not belong in the current MVP.
