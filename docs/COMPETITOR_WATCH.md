# GO IRL Competitor Watch

Status: **current watchlist**
Scope: market monitoring for GO IRL beta and post-beta roadmap
Last updated: 2026-07-08

## Purpose

This document tracks competitors and product signals that can affect GO IRL positioning, MVP scope, and roadmap.

Use it to avoid two mistakes:

1. Building features that competitors already solve better in a different category.
2. Missing useful mechanics that make GO IRL better at converting online intent into real-life attendance.

## Monitoring rule

Monitor competitors for meaningful product changes only:

- new event creation mechanics;
- RSVP / Join / Maybe / Waitlist changes;
- chat or messenger distribution changes;
- host, captain, organizer, coach, trust or verification features;
- launch in Czechia or Central Europe;
- monetization changes;
- Telegram, WhatsApp, Messenger, Instagram or web share improvements;
- local discovery improvements;
- event page or invite UX improvements.

Do not chase visual trends unless they improve the GO IRL core loop.

## Competitor lessons

### Sports apps

Opponent, Squaddler, and Sportual prove that sport is a strong repeatable use case.

Take:

- sport category clarity;
- skill/intensity later;
- capacity;
- event status;
- casual/beginner-friendly labels.

Reject for beta:

- becoming sport-only;
- heavy team tools;
- club management;
- payments.

### Messenger-first organizers

Squaddle is a critical reference because it does not fight the messenger. It uses the messenger context and adds structure.

Take:

- no-download thinking;
- in/out clarity;
- organizer time saving;
- less scrolling through chat.

GO IRL should apply this to Telegram.

### Invite/event page products

Partiful, Luma, and Apple Invites show that event page quality matters.

Take:

- simple event creation;
- good share preview;
- clear RSVP/Join;
- participants/social proof;
- immediate next step after join.

Reject for beta:

- ticketing;
- photo albums;
- playlists;
- party feed mechanics;
- complex guest questionnaires.

### Community/friendship products

Meetup, Meet5, and Timeleft prove that trust and repeat behavior matter.

Take:

- host/captain concept;
- public group safety;
- recurring patterns later;
- newcomer-friendly language.

Reject for beta:

- heavy group creation;
- fully curated matching;
- expensive operations;
- premium membership logic.

### Calendars/ticketing platforms

GoOut, Eventbrite, Olomouc.eu, Olomouc Tourism, Kudy z nudy, and AllEvents already cover the question:

> What is happening in the city?

GO IRL should instead answer:

> Who can I go with, and how do we actually meet?

## Feature import queue

### P0: beta-critical

- clear event cards;
- one-tap Join;
- participant count and limit;
- event chat;
- Telegram share;
- organizer/host visibility;
- browser mock mode;
- stable profile basics.

### P1: after stable beta

- Maybe / Can't go;
- waitlist;
- skill/intensity labels;
- recurring templates;
- better invite page/Open Graph;
- host badges;
- co-host;
- attendance confirmation.

### P2: after market proof

- curated formats;
- city expansion playbook;
- partner venues;
- advanced moderation;
- reputation/trust score;
- optional payments only if there is proven demand and legal/support readiness.

## Hard red lines before beta

Do not build:

- ticketing;
- paid events marketplace;
- subscriptions;
- full club CRM;
- large social feed;
- direct messages;
- AI recommendations;
- photo albums;
- multi-city empty catalog;
- complex profiles;
- public ratings/reviews.

## Monitoring cadence

Weekly competitor monitoring is enough before public launch.

Notify only if there is a meaningful change that affects:

- MVP scope;
- beta risks;
- positioning;
- Telegram/WhatsApp distribution;
- event page expectations;
- host/trust mechanics;
- post-beta roadmap.

No update is needed when competitors only publish generic marketing content.
